# 生产环境创建｜AI与模型部署任务书

## 1. 目标

- 在生产环境完成 AI 大模型（DeepSeek）的本地化部署，搭建 AI Agent 框架，支撑智能问数（Text-to-SQL）与产销平衡优化建议功能。
- 部署运筹优化求解引擎，支撑产销平衡的利润最大化计算。
- 确保模型服务稳定、可监控、可回滚，且数据不出域（满足客户安全要求）。

## 2. 范围

- 在范围内：
  - **DeepSeek 大模型本地部署**
    - 模型版本选择与下载（DeepSeek V3 / R1 / 蒸馏版）
    - 推理框架部署（vLLM / Ollama / TGI 等）
    - GPU 资源配置与显存管理
    - 模型量化策略（如需要，INT4/INT8/FP16）
    - 并发推理与排队策略
  - **AI Agent 框架搭建**
    - 意图识别模块（查询/预警/归因/闲聊分流）
    - Text-to-SQL 模块（Prompt 工程 + Schema 注入 + Few-shot）
    - 安全校验模块（SQL 白名单、行数限制、超时控制）
    - 权限注入模块（按角色追加 WHERE 条件）
    - 自动可视化模块（根据查询结果选择图表类型）
    - 自然语言总结模块
  - **语义字典与知识库**
    - 业务术语→字段映射表维护
    - Few-shot 问答样例库（初始 20-50 个）
    - Doris 表结构注释与 Prompt 模板
  - **运筹优化求解引擎**
    - 求解器安装与配置（PuLP / OR-Tools / 可选 Gurobi）
    - 优化模型服务化封装（API 接口）
    - 模型参数管理（产能/成本/物流矩阵等）
    - 求解日志与结果持久化
  - **模型服务监控与运维**
    - 推理延迟、吞吐量、GPU 利用率监控
    - 模型版本管理与热切换
    - 异常降级策略
- 不在范围内：
  - 模型预训练与微调（使用预训练权重直接部署）
  - GPU 硬件采购（仅提供配置需求，硬件由客户提供）

### 4.2 DeepSeek 部署方案

| 方案 | 模型 | 显存需求 | 推理速度 | 适用场景 |
|------|------|---------|---------|---------|
| **方案A（推荐）** | DeepSeek-V3 67B INT4 | 1×A100 80G 或 2×A30 24G | 快 | 生产环境 |
| **方案B（轻量）** | DeepSeek-V2.5 7B FP16 | 1×RTX4090 24G | 很快 | 演示/小规模 |
| **方案C（中等）** | DeepSeek-V3 67B INT8 | 2×A100 80G | 中等 | 高质量生产 |

> **注意**：具体方案需根据客户实际 GPU 硬件确认。优先保证 Text-to-SQL 的准确率。

### 4.3 运筹优化引擎部署

- 运行环境：Python 3.10+，纯 CPU 即可（无需 GPU）
- 依赖库：
  - `PuLP`：线性规划建模框架
  - `OR-Tools`（Google）：备选求解器，支持 MILP
  - `Gurobi`（可选，商业授权）：大规模问题高性能求解
  - `numpy / pandas`：数据预处理
- 服务化封装：
  - RESTful API 接口（接收参数→构建模型→求解→返回结果）
  - 异步求解支持（复杂问题可排队执行）
  - 求解超时控制（默认 5 分钟，可配置）

## 7. 验收标准

### DeepSeek 模型服务
- 模型加载成功，推理服务稳定运行 ≥72 小时无异常
- 首 Token 延迟 <3 秒，平均生成速度 ≥20 tokens/s
- 并发支持 ≥5 路同时推理
- GPU 利用率、显存监控可看可告警

### AI Agent（智能问数）
- Text-to-SQL 准确率：
  - 高频问题（Top 20）准确率 ≥95%
  - 一般问题准确率 ≥85%
  - 无法处理的问题能正确拒绝并提示
- 端到端响应时间 <10 秒（含 SQL 生成 + Doris 执行 + 结果格式化）
- 权限注入正确（基地用户只查自己基地数据）
- 安全校验有效（DDL/DML 拦截、恶意 SQL 拦截）

### 运筹优化引擎
- 标准场景（10 基地 × 8 区域 × 3 品类）求解时间 <30 秒
- 求解结果可解释（输出约束分析与瓶颈说明）
- 参数变更后可重新求解，结果符合业务逻辑
- What-if 分析功能可用

## 8. 风险与对策

| 风险项 | 影响 | 缓解措施 | 负责人 |
|--------|------|---------|--------|
| 客户无 GPU 或显存不足 | DeepSeek 无法部署 | 准备轻量蒸馏版方案（7B），或考虑 CPU 推理（慢但可用） | 【填写】 |
| Text-to-SQL 准确率不足 | 用户体验差、数据不准 | 充分积累 Few-shot、限定查询范围、完善语义字典 | 【填写】 |
| 模型推理 OOM（显存溢出） | 服务崩溃 | 限制并发数、使用量化版本、监控显存告警 | 【填写】 |
| 优化模型参数缺失 | 无法求解或结果不合理 | 先用估计值/行业经验值，逐步用实际数据替换 | 【填写】 |
| 商业求解器授权问题 | 无法使用 Gurobi | 优先使用开源方案（PuLP/OR-Tools），满足需求即可 | 【填写】 |
| 大模型幻觉/生成错误 SQL | 安全风险 | SQL 白名单 + 只读执行 + 行数限制 + 人工审核兜底 | 【填写】 |

## 9. 降级策略

| 场景 | 降级方案 |
|------|---------|
| DeepSeek 推理服务不可用 | 问数功能临时关闭，显示"AI 助手维护中"提示，核心看板/查询不受影响 |
| Text-to-SQL 生成失败 | 返回兜底提示"抱歉，无法理解该问题，请换个方式提问"，并推荐常用问题 |
| GPU 温度过高 | 自动限流，减少并发推理数 |
| 优化求解超时 | 返回规则引擎的启发式结果作为备选方案，并标注"简化计算结果" |
| 优化参数不完整 | 使用默认参数求解，结果标注"部分参数使用估计值，仅供参考" |
